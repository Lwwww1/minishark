| **成员 B** | 待定 | 协议解析（Ethernet/IP/TCP/UDP/ICMP/DNS/HTTP）、TCP 流重组、TLS SNI 提取、ncurses 终端 UI |

## 二、项目架构概览

```
main.c              — 入口，启动多线程（抓包线程 + 解析/UI 线程）
capture.c / .h      — libpcap 抓包引擎（成员 A）
filter.c / .h       — BPF 过滤器封装（成员 A）
pcap_io.c / .h      — PCAP 文件读写（成员 A）
ring_buffer.c / .h  — 环形缓冲区 / 无锁队列（成员 A）
protocol.c / .h     — 各层协议头结构体定义与解析（成员 B）
tcp_reasm.c / .h    — TCP 流重组（成员 B）
tls_parser.c / .h   — TLS SNI 提取（成员 B）
stats.c / .h        — 流量统计（成员 A）
ui.c / .h           — ncurses 终端界面（成员 B）
common.h            — 公共宏、类型定义（共用）
Makefile            — 编译脚本（共用）
```

| 成员 | 产出 | 提交内容 |

## 六、风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| WSL2 环境网卡抓包权限问题 | 中 | 提前验证 `sudo ./my_sniffer` 可用；备选方案：使用离线 pcap 文件开发解析逻辑 |
| TCP 流重组复杂度超预期 | 高 | 优先实现基础 SEQ 排序 + HTTP 提取；IP 分片重组可降级为简单去重 |
| ncurses 与多线程交互冲突 | 中 | 所有 UI 操作集中在主线程，子线程通过环形缓冲区传递数据 |
| 1Gbps 丢包率难达标 | 中 | 增大抓包缓冲区（`pcap_set_buffer_size`）、使用零拷贝、调整内核参数 |
